# Spec-Drift Review — Nectar Pre-Release Audit

> Category: Notes | Version: 1.0 | Date: 2026-07-02 | Status: Draft
> Scope: `library/knowledge/private/` + `library/knowledge/public/` corpus vs the `src/` implementation, README.md, AGENTS.md, package.json. Deliberately-flagged spec gaps (TLSH confidence thresholds, symbol/directory nectars, review-matches sub-flag syntax) are **not** treated as drift.

## Summary

The low-level implementation is remarkably faithful to the corpus: the two-table DDL, brooding thresholds, cost constants, enricher defaults, prune grace period, port/registry/topology contracts, and the batch prompt are transcribed verbatim and tested. The drift is almost entirely at the **wiring and command-surface layer**, and that is exactly the layer a public user touches first. Three problems dominate. (1) **The mission's middle leg is missing**: the NodeFS watcher, re-association ladder, and cold catch-up are fully implemented (`src/registration/`) but the live daemon never instantiates them — nothing updates identity or descriptions when files change. (2) **Every command the public docs teach is wrong or refuses to run**: the guides say `honeycomb nectar brood` / `review-matches`, the binary is `nectar`, and the mutating `brood`, `prune`, and `review-matches` verbs all exit non-zero by design. (3) **Recall reaches no agent**: the only working recall surface is `nectar search` / `POST /api/hive-graph/search`, while the public docs promise the opposite ("you will not open a Nectar search box... it works through your AI coding assistant") — the Honeycomb recall arm, MCP surface, and harness wiring all live outside this repo and are spec-stage. The private corpus also carries an internal cost contradiction ($2 vs $3.05) and a stale command sweep that ADR-0002 claims was done. The README is honest in places the public docs are not, and stale in the opposite direction (it understates what shipped). Nothing here requires re-architecture; it requires wiring, a command-prefix sweep, and a truth pass over `library/knowledge/public/` before release.

## End-to-end mission check

Mission: *"analyze an entire code base using the brooding process, update it upon change with NodeFS, and recall it as needed."*

### Leg A — Brood an entire codebase: **partially works, not via any documented command** (High)

- The full pipeline (discover → precheck/inherit → bucket → batch/solo describe via Portkey → embed → durable Deep Lake write → projection regen) is implemented for both sync and async stores (`src/brooding/pipeline.ts`, `pipeline-async.ts`) and wired into the live daemon two ways: boot auto-brood (`src/daemon.ts:460-497`) and `POST /api/hive-graph/build` (`src/cli.ts:637-650`, `src/api/daemon-api-wiring.ts`).
- **Gate:** both paths require `~/.deeplake/credentials.json` AND `NECTAR_PORTKEY_ENABLED=1` + API key + config id (`src/cli.ts:611-616`). Neither prerequisite is documented in README or any public doc. Without them the daemon boots, serves `/health`, and does nothing — no warning that brooding is dormant beyond the `/health` provider bits.
- The documented entry points (`honeycomb nectar brood`, `nectar brood`) refuse to run (exit 2). `--dry-run` is the only CLI brood surface that works.

### Leg B — Update on change via the NodeFS watcher: **does not work at all** (Critical)

- `WatchIntake` (node:fs.watch recursive + 500ms debounce, ignore contract, resync-on-null-filename) and `RegistrationService` (stat → classify → 5-step ladder → persist → enrich-queue) are complete and tested (`src/registration/fs-watch.ts`, `service.ts`, `ladder.ts`; `test/registration*.test.ts`).
- **Missing wiring, twice over:** (1) neither `assembleDaemon` nor `runDaemon` constructs a `RegistrationService` — no watcher is ever started; (2) even if one were constructed, `RegistrationService` requires the synchronous `HiveGraphStore`, while the only durable store (`DeepLakeHiveGraphStore`) is async — the sync/async bridge is explicitly deferred (`src/cli.ts:468-477`, `src/hive-graph/store.ts`). The enricher loop therefore only ever drains rows created by a brood; a file edited after the brood keeps its stale description forever, a moved file's recall path goes stale, and copy provenance is never recorded live.
- Cold catch-up is equally unreachable: boot runs projection inheritance only; the ladder never sees the disk diff.

### Leg C — Recall it as needed: **works only through an undocumented surface** (High)

- Working today: `nectar search <query> [--limit N] [--json]` → loopback `POST /api/hive-graph/search` → guarded lexical + vector RRF over latest-described `hive_graph_versions`, with honest `degraded` fallback (`src/cli.ts:423-464`, `src/hive-graph/search.ts`). This surface appears in no public doc and not even in README's CLI section.
- Documented recall (`honeycomb recall ...` README:168-180; "ask your agent" throughout `public/`) depends on the Honeycomb-side 4th arm (PRD-013, out-of-repo, spec-stage) and/or MCP exposure (explicitly deferred, README:237). There is no wiring from any coding agent/harness to Nectar's search endpoint in this repo.
- Within-repo gaps that will degrade recall even once the arm lands: inherited clone rows never re-embed (#8); enricher-cycle projection regen unwired (#19) so committed maps lag descriptions produced by the steady-state loop.

### Cross-leg wiring summary

```
brood (works, gated) ──▶ Deep Lake ──▶ search endpoint (works, undocumented)
        ▲                                   ▲
        │                                   │
watch/ladder (built, NEVER STARTED) ──✂──  enricher (runs, but only sees brood output)
projection: brood-end ✓ · CLI rebuild ✓ · enricher-cycle ✗ (deps not passed)
agent/harness/MCP recall: ✗ (out-of-repo, spec-stage)
```

## Recommended pre-release actions (ordered)

1. Rewrite `library/knowledge/public/` command examples to the verbs that actually run (`nectar brood --dry-run`, `nectar search`, `nectar rebuild-projection`) and soften "automatic freshness" and "through your agent" claims to future tense, or hold the public docs until legs B/C wire up. (Critical)
2. Either wire `RegistrationService` into `runDaemon` behind the sync/async bridge, or add an explicit "live watch is not yet enabled" notice to README/`/health`. (Critical)
3. Sweep the `honeycomb nectar` prefix out of the private corpus per ADR-0002's own claim; fix `overview.md:67` ($2 → ~$3). (High)
4. Re-embed inherited projection rows on first boot (or mark them so the enricher picks them up) per `portable-registry.md:85`. (High)
5. Pass `projectionWriter`/`projectionStore` into the live enricher wiring so trigger #2 fires. (Medium)
6. Refresh AGENTS.md implementation-status block and README's CLI/status sections (both stale, in opposite directions). (Medium)
7. Document the brood prerequisites (`~/.deeplake/credentials.json`, `NECTAR_PORTKEY_*`) wherever brooding is promised. (Medium)

*Deliberate spec gaps confirmed and not flagged: TLSH confidence thresholds (unspecified on purpose, `review-cli.ts` honors this), symbol/directory nectars (v2, `kind` column reserved), review-matches accept/reject flag grammar (injected-decider seam, no invented flags).*
