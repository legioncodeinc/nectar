# Source Graph: User Stories

> Category: Data | Version: 1.0 | Date: June 2026 | Status: Draft

The engineering and operator contracts that the `source_graph` / `source_graph_versions` schema imposes, written as testable user stories with acceptance criteria. Personas: the schema designer, the Deep Lake client writing rows, the recall query reading rows, the heal pass at daemon boot, and the operator auditing tenancy isolation.

## Scope and persona key

These stories are engineering scope: they describe what the schema must support and what the code that reads and writes it must guarantee. They are not product requirements and they do not specify UI, cost, or rollout — those belong in PRDs. Each story names a persona and a contract; each acceptance criterion is a checkable property of the schema or the code that touches it.

- **Schema designer** — owns the `ColumnDef` arrays and the catalog group; reasons about table shape.
- **Deep Lake client (writer)** — the hiveantennae worker code that mints nectars and appends version rows.
- **Recall query (reader)** — the hybrid recall arm that reads `source_graph_versions`.
- **Heal pass** — the `withHeal` code path at daemon boot.
- **Operator** — the human auditing tenancy, running rebuilds, or investigating isolation.

## Two-table split rationale

**US-SG-001** — As a schema designer, I separate file identity from file content history into two tables so that identity stays stable while content changes freely. **Acceptance criteria:** (a) `source_graph` carries no content columns (`content_hash`, `path`, `description`, `embedding` are absent); (b) `source_graph_versions` carries no identity-only columns (`derived_from_nectar`, `fork_content_hash`, `kind` are absent); (c) the nectar column appears on both as the join key.

**US-SG-002** — As a schema designer, I keep the identity row immutable across content edits so that the primary key never churns. **Acceptance criteria:** (a) a content edit appends to `source_graph_versions` only; (b) the `source_graph` row for that nectar is not rewritten (only `last_update_date` moves); (c) the nectar value is byte-identical before and after the edit.

---

## Nectar as immutable primary key

**US-SG-003** — As a Deep Lake client writing rows, I mint a nectar exactly once per logical file and never recompute it, so that identity survives daemon upgrades. **Acceptance criteria:** (a) the nectar is a 26-char ULID generated by the minting call; (b) no code path recomputes `source_graph.nectar` from file properties (path, content, mtime); (c) a daemon upgrade that changes minting logic leaves existing nectars untouched.

**US-SG-004** — As a Deep Lake client writing rows, I write the `source_graph` identity row before any `source_graph_versions` row that references it, so that the foreign key is never violated. **Acceptance criteria:** (a) for a newly-minted nectar, the identity-row INSERT precedes the version-row INSERT in the same transaction; (b) no version row exists whose `nectar` has no matching `source_graph` row; (c) a partial write that inserts the version row without the identity row is rejected.

---

## Append-only versions contract

**US-SG-005** — As a Deep Lake client writing rows, I append a new version row on every observed content change rather than updating an existing row, so that the history chain is complete. **Acceptance criteria:** (a) an edit that changes content produces a new `source_graph_versions` row with a new `content_hash` and `seq = prev_seq + 1`; (b) the prior version row is retained unchanged; (c) no code path updates an existing version row's `content_hash` or `path` in place.

**US-SG-006** — As a Deep Lake client writing rows, I leave the description columns empty and `describe_status = 'pending'` at version-row insert, so that enrichment is decoupled from observation. **Acceptance criteria:** (a) a newly-appended version row has empty `title`, empty `description`, `'[]'` `concepts`, null `embedding`, and `describe_status = 'pending'`; (b) the enricher later sets these columns and flips status to `'described'`; (c) recall does not surface the row until status is `'described'`.

---

## Composite key `(nectar, content_hash)`

**US-SG-007** — As a Deep Lake client writing rows, I treat `(nectar, content_hash)` as the version-row identity, so that re-observing unchanged content is a no-op. **Acceptance criteria:** (a) before appending a version row, the client checks whether `(nectar, content_hash)` already exists; (b) if it exists, no row is written and no enrich job is enqueued; (c) a save that changes no bytes produces zero new rows.

**US-SG-008** — As a Deep Lake client writing rows, I detect the copy event by checking whether a new path's content hash matches another nectar's latest-version content hash, so that copy-paste becomes a provenance edge. **Acceptance criteria:** (a) when a new path's content hash matches an existing nectar's latest version, the daemon mints a fresh nectar for the new path; (b) the new nectar's `source_graph.derived_from_nectar` is set to the source nectar; (c) `fork_content_hash` records the source's content at the fork point; (d) both nectars coexist as distinct identities.

---

## `seq` monotonic counter

**US-SG-009** — As a recall query reading rows, I select the latest version per nectar by `MAX(seq)` rather than by timestamp parsing, so that "current state" is a cheap deterministic query. **Acceptance criteria:** (a) the latest-per-nectar subquery uses `MAX(seq) GROUP BY nectar`; (b) `seq` is monotonically increasing per nectar (0, 1, 2, …); (c) no two version rows for the same nectar share a `seq` value.

**US-SG-010** — As a Deep Lake client writing rows, I compute the new version's `seq` as the prior maximum `seq` for that nectar plus one, so that ordering is preserved across concurrent writers. **Acceptance criteria:** (a) the `seq` assignment reads the current max for the nectar within the same transaction as the insert; (b) two concurrent edits to the same nectar produce distinct, ordered `seq` values, not a collision.

## Tenancy columns

**US-SG-011** — As a schema designer, I carry explicit `org_id`, `workspace_id`, and `project_id` on both tables so that the versions table is queryable in isolation for recall. **Acceptance criteria:** (a) both tables have all three tenancy columns; (b) the versions table does not require a join to `source_graph` to resolve tenancy; (c) a recall query scopes by all three columns in its `WHERE` clause.

**US-SG-012** — As an operator auditing tenancy isolation, I verify that every recall query carries the tenancy predicate, so that one project's file descriptions never leak into another project's results. **Acceptance criteria:** (a) the recall arm's SQL includes `org_id = :org AND workspace_id = :workspace AND project_id = :project`; (b) no recall code path omits the predicate; (c) a query scoped to project A returns zero rows from project B even when both projects describe files with identical paths.

---

## Cross-agent visibility

**US-SG-013** — As a schema designer, I omit `agent_id` and `visibility` columns from both Hivenectar tables, so that file identity and description are shared across all agents in the project. **Acceptance criteria:** (a) neither table has an `agent_id` column; (b) neither table has a `visibility` column; (c) recall queries against these tables carry no `agent_id` predicate (contrast with `sessions`/`memory`, which do).

**US-SG-014** — As an operator, I confirm that two agents working in the same project see the same file descriptions, so that semantic recall is consistent across the team. **Acceptance criteria:** (a) a file described by agent A's enricher is visible to agent B's recall without any per-agent grant; (b) a fresh teammate's clone inherits the described rows through cloud sync plus the projection; (c) isolation is org→workspace→project only, with no per-agent subdivision.

---

## `describe_status` lifecycle

**US-SG-015** — As a recall query reading rows, I filter to `describe_status = 'described'`, so that undescribed files do not pollute semantic results. **Acceptance criteria:** (a) the recall arm's subquery includes `WHERE describe_status = 'described'`; (b) rows in `pending`, `failed`, `skipped-too-large`, or `skipped-binary` states are excluded; (c) a file described then re-edited surfaces its newly-described latest version, not its prior described version.

**US-SG-016** — As a Deep Lake client writing rows, I transition `describe_status` through the documented lifecycle so that the enricher can resume after failures. **Acceptance criteria:** (a) new version rows start at `'pending'`; (b) successful enrichment sets `'described'` and fills `title`/`description`/`embedding`/`described_at`/`describe_model`; (c) a failed enrichment sets `'failed'` and the row is retried on the next lazy pass, returning to `'pending'` or advancing to `'described'`.

**US-SG-017** — As a Deep Lake client writing rows, I set `describe_status` to the appropriate skip value for files the enricher cannot process, so that recall excludes them cleanly and the skip is auditable. **Acceptance criteria:** (a) a file exceeding the describe limit gets `'skipped-too-large'`; (b) a binary file with no extractable text gets `'skipped-binary'`; (c) both skip values are excluded by the recall `described` filter.

---

## Lazy schema healing

**US-SG-018** — As a heal pass at daemon boot, I additively `ALTER` any Hivenectar table missing a column that the current daemon expects, so that upgrades converge without manual migration. **Acceptance criteria:** (a) the heal pass is invoked through `withHeal` against the catalog group; (b) only additive `ALTER TABLE ... ADD COLUMN` statements are issued; (c) existing rows receive the column's default (e.g. `'[]'` for `concepts`); (d) no column is dropped, renamed, or type-changed by a heal.

**US-SG-019** — As a schema designer, I define each table's columns exactly once in the daemon's schema module `ColumnDef` array and register it with the catalog group, so that hand-rolled ALTERs are never needed. **Acceptance criteria:** (a) the `ColumnDef` array is the single source of truth for each table's columns; (b) no code path issues a raw `ALTER TABLE` against `source_graph` or `source_graph_versions` outside the heal pass; (c) a hand-rolled ALTER is treated as a defect, not a supported operation.

## Projection contract

**US-SG-020** — As a Deep Lake client writing rows, I regenerate the `.honeycomb/nectars.json` projection from `source_graph_versions` after every brood and enrich cycle, so that the projection stays a faithful derived view. **Acceptance criteria:** (a) Deep Lake writes complete before the projection write begins; (b) the projection contains the latest described version per nectar, scoped to the project; (c) the projection write is atomic (temp file plus rename); (d) a crashed regeneration leaves the prior projection intact.

**US-SG-021** — As an operator, I can rebuild the projection from Deep Lake alone via `honeycomb hivenectar project --rebuild-projection`, so that a lost or corrupt projection is recoverable without re-brooding. **Acceptance criteria:** (a) the rebuild command scans `source_graph_versions` and produces the projection with no other inputs; (b) the rebuilt file is byte-identical to a fresh generation modulo `generated_at`; (c) if the rebuild could not reproduce the projection from Deep Lake alone, that would indicate a sidecar violation and is a defect.

**US-SG-022** — As a schema designer, I ensure the projection is never a source of truth, so that FR-8 (no sidecars) is satisfied. **Acceptance criteria:** (a) no read path consults the projection as authoritative state (recall reads Deep Lake directly); (b) the projection flows into Deep Lake only on fresh-clone inheritance, and only for nectars local Deep Lake lacks; (c) a hand-edit to the projection is overwritten on the next regeneration.

---

## Path mutation across versions

**US-SG-023** — As a Deep Lake client writing rows, I record a file move by appending a version row with the new path under the same nectar, so that the rename is captured in the history chain. **Acceptance criteria:** (a) a move produces a new `source_graph_versions` row for the existing nectar with the new `path` and incremented `seq`; (b) the prior version row retains its old path as history; (c) the `source_graph` identity row is not rewritten for a pure move (only `last_update_date` moves); (d) recall surfaces the file under its current path, not its historical paths.

---

## Identity stability across re-association

**US-SG-024** — As a Deep Lake client writing rows, I carry an existing nectar across a move-and-edit via the re-association ladder rather than minting a new one, so that history continuity is preserved. **Acceptance criteria:** (a) an exact content-hash match to a missing file (ladder step 3) reuses the nectar and appends a version row; (b) a fuzzy TLSH match above the confidence threshold (step 4) reuses the nectar and flags confidence; (c) a fuzzy match below threshold mints a new nectar rather than risking a mis-association; (d) a nectar, once minted, is never reused for a different logical file.

**US-SG-025** — As a Deep Lake client writing rows, I never delete a nectar during re-association, so that the history chain is trustworthy. **Acceptance criteria:** (a) a file with no re-association candidate (step 5) mints a new nectar rather than scanning for orphans to reuse; (b) a deleted file's nectar remains in Deep Lake as history; (c) nectar removal is a separate explicit human-triggered operation (`prune --confirm`), not a side effect of re-association.
